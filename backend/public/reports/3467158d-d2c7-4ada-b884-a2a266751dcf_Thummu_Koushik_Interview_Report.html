📚 What a Good Answer Looks Like
localStorage: persists indefinitely even after closing and reopening the browser — cleared only via JavaScript or browser settings. Scoped to origin. Not sent to server. Max ~5MB. sessionStorage: cleared when the tab is closed. Each tab has its own sessionStorage — not shared across tabs. Not sent to server. Same ~5MB limit. Cookies: sent to the server automatically with every HTTP request (via Cookie header). Can be set with expiry, HttpOnly (not accessible via JS — safer for auth tokens), Secure (HTTPS only), and SameSite (CSRF protection). Max ~4KB.

Understands core concepts but misses depth — knows what inline and internal CSS are Could not fully explain the specificity implications or when each should (and should not) be used

📚 What a Good Answer Looks Like
Inline CSS: styles applied directly on an element via the style attribute — e.g. <p style="color:red">. Highest specificity; overrides internal and external styles. Avoid in production — hard to maintain, not reusable. Internal CSS: styles defined inside a <style> tag in the <head>. Scoped to that HTML page only. Useful for single-page styles; not reusable across pages. External CSS (separate .css file linked with <link>) is preferred for maintainability and caching.

✗ Weak — Async concept roughly known; Promises and async/await not understood with sufficient depth for a full stack interview
📚 What a Good Answer Looks Like
A Promise is an object representing the eventual completion or failure of an async operation. Three states: pending (initial), fulfilled (resolved with a value), rejected (failed with a reason). Chain with .then(value => ...) for success and .catch(err => ...) for errors. Avoids callback hell. async/await: syntactic sugar over Promises. An async function always returns a Promise. await pauses execution inside the function until the Promise resolves — making async code read like synchronous code. Wrap in try/catch for error handling. Example: const data = await fetch(url).then(r => r.json()).

Database Indexing — What is indexing and what is the trade-off?
Explanation was weak — audit flagged database indexing as a specific gap Could not clearly explain what an index is, what data structure it uses, or why write operations are slowed
✗ Weak — Indexing purpose and trade-off not clearly explained
📚 What a Good Answer Looks Like
An index is a data structure (typically B-tree in SQL, also B-tree in MongoDB) built on one or more fields so the database can find documents/rows without scanning the entire collection/table. Dramatically speeds up read queries (SELECT WHERE, find() with filters). Trade-off: every INSERT, UPDATE, and DELETE must also update all indexes on that collection — adding overhead to write operations. Rule of thumb: index fields you query frequently; avoid over-indexing collections with heavy write loads. Example: db.tasks.createIndex({'{'} userId: 1 {'}'}) speeds up all queries filtering by userId.
