📚 What a Good Answer Looks Like
Sort jobs by difficulty keeping profit aligned: jobs = sorted(zip(difficulty, profit)). Create prefix max profit array: max_profit[i] = max(max_profit[i-1], jobs[i][1]). Sort workers. Use two pointers: for each worker (sorted), advance job pointer while difficulty ≤ ability, track max profit seen. Add that max to total. Time: O(n log n + m log m), Space: O(n).

📚 What a Good Answer Looks Like
Two cases: (1) Max subarray is non-wrapping → standard Kadane's algorithm gives max_normal. (2) Max subarray wraps → equivalent to total_sum - min_subarray (using Kadane's for minimum). Answer: max(max_normal, total_sum - min_subarray). Edge case: if all elements are negative, return max_normal (avoid returning 0 when min subarray = total). Time: O(n), Space: O(1).

An index (B-Tree by default) lets the database jump directly to matching rows instead of scanning the full table — dramatically speeds up SELECT/WHERE/JOIN. 📚 What a Good Answer Looks Like
Atomicity: all-or-nothing — if any part of a transaction fails, the whole thing rolls back. Consistency: database moves from one valid state to another — all constraints satisfied. Isolation: concurrent transactions don't see each other's partial changes. Durability: once committed, data survives system failures (persisted to disk). These guarantee reliable transactions in relational databases.

📚 What a Good Answer Looks Like
TRUNCATE: removes ALL rows, keeps table structure (schema, indexes). Fast DDL — usually not rollback-able. DROP: removes the entire table permanently — structure, data, indexes, triggers. DDL.
